프로젝트 개요

스마트 타이어 프로젝트는 차량 타이어의 상태를 실시간으로 모니터링하여 안전성을 높이기 위해 시작되었습니다. 압전 소자 센서를 활용하여 타이어의 진동과 압력 변화를 감지하고, 이를 기반으로 타이어 상태를 분석하여 스마트폰 앱으로 알림을 제공하는 것이 목적입니다.

제작의 용이성을 위해 자전거 바퀴로 프로토타입을 만듭니다.

메인 아이디어

압전 센서를 이용하여 실시간으로 데이터를 수집하고, 수집된 데이터를 전처리하여 무선으로 서버에 전송합니다. 이 데이터를 머신러닝을 통해 분석하여 스마트폰 앱으로 알림을 제공하는 시스템입니다. 주요 하드웨어로는 Murata 7BB-20-3 압전 센서, ESP32-C3 Arduino 모듈, Raspberry Pi 4가 사용되었습니다.

이 시스템은 타이어 마모 및 이상 진동을 조기에 감지하여 사고를 예방하고, 사용자가 타이어 상태를 쉽게 모니터링할 수 있도록 지원합니다.

하드웨어 구성

주요 하드웨어 구성:

  • Murata 7BB-20-3: 압력과 진동 감지
  • ESP32: 센서 데이터 수집 및 전처리
  • Raspberry Pi 4: 데이터 분석 및 시각화

부품 부착 방법

글루건으로 붙인 부분은 전기가 통하지 않습니다. 글루건의 주요 성분은 전기를 전도하지 않는 물질로 구성되어 있어 절연체 역할을 합니다. 따라서 간단한 전선 고정이나 절연 목적으로 사용할 수 있습니다.

타이어가 끼워지는 림 부분에 압전 센서를 글루건으로 고정하고, 스포크 부분에 ESP32 모듈을 글루건이나 3D프린터 등으로 제작한 케이스로 고정합니다.

안장 아래 부분 등 구르지 않는 부분에 라즈베리파이 보드를 고정합니다.

ESP32 및 센서

ESP32는 센서 데이터를 수집하고 전처리를 하여 서버로 데이터를 무선 전송합니다.

센서 결선

ESP32는 6개의 아날로그 12bit 입력을 가집니다. 12bit 데이터이면 우리 프로젝트에 충분한 분해능을 제공하고 4개의 센서를 고려하고 있었기에 6개의 자체 ADC를 가지고 있는 이 아두이노 모델이 프로젝트에 적합하다고 봅니다.

압전센서는 무라타 7BB 시리즈의 20-3 압전센서를 사용합니다.

센서를 테스트한 결과 지속 시간이 짧아 정확하지는 않지만 -4V에서 15V 정도의 전압이 출력되었습니다. 자료를 추가로 찾아보았을 때 해당 시리즈의 압전 센서는 수십 V의 전압을 만든다고 합니다. 또 우리가 사용하는 환경과 같은 진동을 측정하는 상황에서는 수백 mV이내의 신호를 만듭니다.

ESP32는 입력이 3.3V를 초과하는 경우 파손이 발생하고, 0.3V에서 2.8V 사이 전압에서 ADC 동작을 권장하고 있습니다.(이를 벗어난는 범위에서는 오차/포화 문제 발생)

따라서 오프셋 전압을 추가하여 범위를 입력가능한 전압 영역으로 가져오고 입력 전압을 3.3V로 제한하기 위해 온도 특성도 고려하여 제너 전압이 3.2V인 제너다이오드와 음전압 스파이크로 부터 보호하기 위해 쇼트키 다이오드를 사용합니다.

위 회로는 입력 신호가 ADC의 입력 범위를 벗어나지 않도록 DC 전압을 추가하고 비반전 증폭기로 신호를 증폭하여 ADC 입력 범위에 맞추어 범위를 최대한 활용합니다.

시뮬레이션 결과 -500mV~500mV의 신호를 우리가 원하던 0.3V에서 2.8V로 가져올 수 있었습니다. 피드백 저항과 오프셋 전원으로의 전압분배 저항의 값을 조절하여 게인과 오프셋 전압을 조절하여 원하는 위치에서 동작할 수 있습니다.

ESP32 배터리 결선

배터리 충전 결선은 위와 같이 합니다. TP4056 리튬배터리 충전모듈을 사용하며 3.7V, 1000mAh의 리튬 폴리머 배터리를 사용합니다.

ESP32의 VUSB단자와 충전 모듈의 충전 전원 +단지에 연결하여 ESP32에 USB가 연결되었을 때 충전 모듈에 전원이 공급될 수 있도록 합니다. ESP32의 USB포트로 충전과 코드 업데이트를 모두 할 수 있습니다.

위의 결선은 ESP32의 Vin 단자가 VUSB 단자와 전원부에 다이오드를 통해 연결되었기에 가능합니다.

라즈베리파이

라즈베리파이는 데이터 분석과 앱으로 데이터를 전송하는 역할을 합니다.

라즈베리파이의 전원 공급은 리튬 배터리를 통해서 합니다. 18650 리튬 배터리는 원통형의 배터리로 재충전이 가능합니다. USB단자가 있는 배터리 홀더를 통해 단순하게 전력을 공급 받습니다.

18650 배터리의 용량은 2000mAh에서 3500mAh 사이로 전력 사용을 3W 정도로 계산을 하여도 3시간 이상 사용이 가능합니다.

배터리 홀더는 4구, 네 개의 리튬전지가 들어가는 홀더를 사용해 10,000mAh에 가까운 용량을 계획하고 있습니다.

소프트웨어 및 데이터 처리

ESP32는 압전 센서로부터 데이터를 수집하고, 이를 전처리하여 MQTT 프로토콜을 통해 서버로 전송합니다. 서버에서는 머신러닝 모델을 사용하여 타이어 상태를 분석하고, 결과를 사용자에게 시각적으로 제공하는 시스템입니다.

데이터 수집 주기

데이터 수집 주기를 결정하는데 두 가지 고려사항이 있습니다. 하나는 알고리즘에서 데이터를 처리하는 시간보다 길어야 할 것이며, 다른 하나는 자전거의 속도를 고려하였을 때 충분한 데이터를 얻을 수 있어야 할 것입니다.

데이터 처리 관점

무선 전송 시간은 전송할 데이터의 크기와 연결된 네트워크의 속도에 따라 달라집니다.

총 예상 처리 시간: 각 과정에 대한 예상 처리 시간을 합산하면

  • 필터: 50-100μs
  • 정규화: 10-20μs
  • 피크 감지: 10-30μs
  • 무선 전송: 2-5ms

총 소요 시간: 3ms 이내

데이터 처리의 소요 시간은 수 밀리초 정도로 안정성을 위해 수집 주기를 10ms 이상으로 합니다.

자전거 속력 관점

자전거 속력을 20km/h로 가정하면(한 바퀴에 얻을 수 있는 데이터가 적은 경우를 가정하기 위해 다소 빠른 속력을 가정)

20 km/h = 20,000m ÷ 3,600s ≈ 5.56m/s

타이어 반지름 r = 0.35m

타이어의 둘래는 2πr ≈ 2.2m

타이어 회전수 f = 5.56m/s ÷ 2.2m ≈ 2.53Hz

타이어 회전 주기 T ≈ 1 ÷ 2.53Hz ≈ 0.395s

100ms마다 데이터를 수집할 경우, 타이어가 한 바퀴 도는 395ms 동안 약 4회의 데이터를 수집할 수 있습니다. 한 바퀴 회전 시 여러 지점에서 데이터를 측정 가능하며, 자전거 속도 변화나 타이어 진동 패턴을 잘 반영할 수 있습니다. 따라서 수집주기는 100ms 이하가 되도록 합니다.

바퀴 회전 주기에 기반한 진동 성분은 1.27Hz(10km/h)에서 2.5Hz(20km/h) 사이에서 나타날 것입니다.

여러 연구와 실험에서 자동차와 같은 차량의 타이어 진동은 1~30Hz 사이의 저주파 대역이 대부분인 것으로 나타났습니다.

자전거의 경우도 유사하게, 지면의 요철이나 충격이 타이어와 프레임을 통해 전달되는 과정에서 저주파 대역이 주요한 신호가 될 것입니다.

노면의 요철에 의해 발생하는 진동 주파수는 타이어의 회전 주파수와 그 배수에 의해 제한되기 때문에 주로 1~30Hz 내외의 주파수 범위를 넘지 않을 겁니다.

이에 대해서 실험을 통해 FFT 분석을 통해 주로 진동성분이 어디에 집중되는지 알아보면 좋을 것 같습니다.

샘플링 과정에서 고주파 영역은 상당히 필터링이 됩니다. 우리는 저역 통과 필터의 필요성을 재검토하게 되었고, 우리가 필요한 주파수 대역은 수 Hz 정도로 예상되며 여전히 저역 통과 필터가 필요할 수 있으나 우선 필터링 과정을 제외하고 필요하다면 추가하기로 하였습니다.

데이터 전처리

수집된 데이터는 노이즈 제거와 정규화 과정 등의 전처리 과정을 거칩니다.

고려한 전처리 과정은 노이즈 제거와 정규화 피크 감지, 이상값 제거 등이 있었습니다. 노이즈 제거와 정규화는 머신러닝 과정에서 학습 효율과 분석 효율성 측면에서 필수라고 보았고, 피크 감지는 머신 러닝 모델이 충분히 판단할 수 있는 부분이지만 마찬가지로 학습과 분석 효율에 도움이 되리라 보았습니다. 이상값 제거 또한 한다면 더 효율적인 학습과 분석을 할 수 있으리라 생각하였지만 필요한 데이터 또한 제거될 수 있을 것 같다는 점과 너무 많은 전처리 과정을 가지는 것은 처리 시간 관점에서 좋지 않다고 판단하여 아래 세 가지 전처리 과정만 가집니다. 후에 계산해 본 결과 소요 시간에는 여유가 있음을 확인하였습니다.

이동 평균 필터

필터: 이동 평균 필터와 저역 통과 필터를 고려하였으며 구현이 쉬운 이동 평균 필터로 노이즈 제거를 합니다.

정규화

정규화: 최소 -1에서 최대 1로 정규화합니다.

피크 감지

피크 감지: 특정 값을 초과하는 경우를 피크로 판단하고 피크가 발생한 것을 하나의 이벤트로 데이터에 추가합니다. 정규화 이후 이므로 0.8이상을 피크로 판단하는 등으로 만들 수 있습니다.

전처리 함수 테스트

통신 방식 검토

  • ESP-NOW
  • MQTT over WiFi
  • UDP over WiFi
  • BLE(Bluetooth Low Energy)

1. ESP-NOW

장점:
- 매우 낮은 지연 시간 (약 1ms)
- 간단한 설정과 사용
- 낮은 전력 소비

단점:
- 라즈베리 파이와 직접 통신 불가
- 제한된 데이터 크기 (최대 250바이트)

구현 플랜:
1. ESP32에서 ESP-NOW로 데이터 전송
2. 중간 ESP32를 사용하여 데이터 수신
3. 중간 ESP32에서 라즈베리 파이로 UART/SPI를 통해(유선으로) 데이터 전달
4. 데이터를 모아서 전송하는 경우, 250바이트 단위로 나누어 전송

2. MQTT over WiFi

장점:
- 안정적인 데이터 전송
- 확장성이 좋음
- 다양한 QoS 레벨 제공

단점:
- ESP-NOW보다 상대적으로 높은 지연 시간
- WiFi 연결 필요

구현 플랜:
1. 라즈베리 파이를 MQTT 브로커로 설정
2. 긱블미니 ESP32에서 MQTT 클라이언트 구현
3. QoS 0 사용하여 빠른 전송 구현
4. 데이터를 모아서 전송할 경우, 적절한 크기의 배치로 구성하여 전송

3. UDP over WiFi

장점:
- 낮은 지연 시간
- 간단한 프로토콜
- 대량 데이터 전송에 적합

단점:
- 데이터 손실 가능성
- WiFi 연결 필요

구현 플랜:
1. 라즈베리 파이에서 UDP 서버 구현
2. 긱블미니 ESP32에서 UDP 클라이언트 구현
3. 데이터를 모아서 전송할 경우, MTU 크기를 고려하여 패킷 구성
4. 필요시 간단한 재전송 메커니즘 구현

4. BLE (Bluetooth Low Energy)

장점:
- 매우 낮은 전력 소비
- 안정적인 연결
- 라즈베리 파이와 직접 통신 가능

단점:
- 상대적으로 낮은 데이터 전송 속도
- 제한된 범위 (약 100m)

구현 플랜:
1. 긱블미니 ESP32에서 BLE 서버 구현
2. 라즈베리 파이에서 BLE 클라이언트 구현
3. GATT 프로파일을 사용하여 데이터 전송
4. 데이터를 모아서 전송할 경우, MTU 크기(보통 20-23바이트)를 고려하여 여러 패킷으로 나누어 전송

각 방법의 권장 사용 시나리오:

1. ESP-NOW: 매우 빠른 응답 시간이 필요하고, 추가 ESP32를 중간에 사용할 수 있는 경우
2. MQTT over WiFi: 안정적인 데이터 전송이 필요하고, 시스템 확장성을 고려해야 하는 경우
3. UDP over WiFi: 빠른 전송 속도와 낮은 지연 시간이 필요하며, 약간의 데이터 손실이 허용되는 경우
4. BLE: 전력 소비가 중요하고, 데이터 전송 속도가 상대적으로 낮아도 되는 경우

최종 선택은 구체적인 요구사항(전송 속도, 안정성, 전력 소비 등)과 실제 테스트 결과를 바탕으로 결정합니다.
